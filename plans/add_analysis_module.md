# План добавления модуля аналитики

## Ключевые метрики и аналитические срезы

Для построения полезной аналитики необходимо оперировать следующими показателями (с учётом специфики):

### Поток и объёмы (Throughput)

- Количество закрытых сущностей за период (день/неделя/месяц) в разрезе типов (заявки, задачи, работы, планы).
- Количество созданных сущностей за период – для сравнения с закрытыми.
- Накопленное отставание (создано минус закрыто нарастающим итогом) – показывает рост или сокращение бэклога.

### Активная работа (WIP – Work In Progress)

- Количество задач в статусе «В работе» на текущий момент (общее и по исполнителям).
- Среднее время нахождения задачи в статусе «В работе» (для выявления «зависших» задач).
- Распределение WIP по типам – не завалили ли команду срочные заявки в ущерб плановым задачам.

### Сроки и SLA

- Среднее время закрытия заявок (в календарных или рабочих днях).
- Доля заявок, закрытых с нарушением SLA (более 2 рабочих дней) – по исполнителям, по тематикам.
- Время цикла (Cycle Time) для задач: разница между датой создания и датой закрытия.
  Важно: интерпретировать с учётом возможного длительного простоя в бэклоге. Для более точной оценки можно отдельно считать время для задач, которые были в статусе «В работе».
  Время ожидания начала работы – если фиксируется дата первого перехода в «В работе»: среднее время от создания до старта.

### Баланс типов работ

- Соотношение закрытых сущностей по типам (заявки, задачи, работы, планы) за период.
  Инсайт: не перевешивает ли операционка (заявки+работы) над стратегическими задачами.
- Динамика структуры – как меняется баланс неделя к неделе.

### План/факт (для недельных планов)

- Процент выполненных плановых задач (количество закрытых из числа запланированных на неделю).
- Сравнение запланированных и фактически выполненных объёмов по типам.

### Бэклог и «долгострои»

- Количество невыполненных задач со статусом «Новая» или «Отложено» (бэклог).
- Возраст задач в бэклоге – самые старые задачи, требующие ревизии.
- Список задач, находящихся в статусе «В работе» дольше нормативного срока (например, более 30 дней).

### Индивидуальная результативность

- Количество закрытых задач/заявок по каждому сотруднику (в разрезе типов).
  Осторожно: учитывать разную сложность, но устойчивые аномалии – повод для разговора.
- Соблюдение индивидуального SLA (для заявок) – у кого чаще просрочки.

## Предварительные этапы

- SLA заявок
  - Решить вопрос с расписаниями для JS -> задача по "компиляции расписания инвентори в JS"
  - Организовать оценку времени выполнения заявок согласно расписания
- Использовать для задач отдельно "запланировано на" и "крайний срок" (Непонятно что в таком случае делать с крайними сроками. )
- Модуль анализ состояния: Предлагать с утра провести анализ (при отказе запрашивать повторно через час, отображать значок ожидания). Взять в работу и отложить задачи согласно анализа:
  - При наступлении запланированного срока задачи -> предлагать брать в работу, либо сдвигать вправо
  - Если по задаче в работе не было оформлено работ и не было добавлено комментариев, то предлагать внести работу или сдвинуть вправо/отложить (можем ли мы понять была ли задача в работе вчера?)
Разделить долгий ящик на три (в порядке убивания приоритета): ожидает выполнения, новая (бэклог), отложено
Добавить пометки к дням, чтобы можно было обозначать внеплановые ситуации

## Ручной режим

- Выбирать одну задачу и согласовывать по ней крайний срок. Контролировать соблюдение этого срока. Если пользователь не может соблюсти срок, то предлагать согласовать новый. Если пользователь срывает срок без согласования это прямо косяк.
