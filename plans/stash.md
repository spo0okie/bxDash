# Долгий ящик

## Цель

В приложении есть такая секция для задач как "долгий ящик" - задачи без крайнего срока, либо с крайним сроком больше, чем граница отображаемого календаря.
Нужно разделить долгий ящик на три (в порядке убивания приоритета) по статусу задачи: ожидает выполнения, новая (бэклог), отложено.

### UI layout

Новый режим отображения должен быть опциональным, чтобы можно было переключаться между ним и старым режимом, в котором все задачи без крайнего срока отображаются в одном "долгом ящике".

## Архитектурные изменения

### Изменения в данных модели

Для реализации разделения долгого ящика на три приоритета (низкий, средний, высокий) необходимо внести следующие изменения в структуру данных:

1. **Добавление приоритета в элемент**: В класс `DashItem` добавить новое свойство `priority` типа number (0-2) с геттером и сеттером, где:
   - `0` — низкий приоритет
   - `1` — средний приоритет
   - `2` — высокий приоритет

2. **Инициализация приоритета для существующих элементов**:
   - Для задач (`TaskItem`): приоритет определяется из существующего атрибута `priority` (если присутствует) или устанавливается в `1` (средний) по умолчанию.
   - Для работ (`JobItem`): приоритет всегда устанавливается в `1` (средний) согласно требованию.
   - Для заявок (`TicketItem`): приоритет всегда устанавливается в `2` (высокий) согласно требованию.
   - Для планов (`PlanItem`) и записок (`MemoItem`): приоритет определяется из их специфических атрибутов или устанавливается в `1` (средний) по умолчанию.

3. **Изменение логики фильтрации в bucket**: В методе `filterItem` класса `IntervalItem` (для bucket-интервалов с `end === null`) добавить проверку приоритета элемента (`priority`) для определения, в какой из трех секций долгого ящика он должен отображаться.

4. **Обновление computed-свойств**: В хранилищах элементов добавить computed-свойства для фильтрации элементов по приоритету bucket:
   - `lowPriorityItems` — элементы с `priority === 0`
   - `mediumPriorityItems` — элементы с `priority === 1`
   - `highPriorityItems` — элементы с `priority === 2`

### Изменения в логике drag-n-drop

Для корректной работы с разделенным долгим ящиком необходимо обновить логику drag-n-drop:

1. **При перетаскивания элемента в секцию долгого ящика**: Помимо установки `deadline = null`, необходимо также устанавливать соответствующий `priority` в зависимости от секции, в которую был перемещен элемент:
   - В раскладке "внизу" (все пользователи):
     * Верхняя секция → `2` (высокий приоритет)
     * Средняя секция → `1` (средний приоритет)
     * Нижняя секция → `0` (низкий приоритет)
   - В раскладке "справа" (выбранный пользователь):
     * Левая секция → `2` (высокий приоритет)
     * Средняя секция → `1` (средний приоритет)
     * Правая секция → `0` (низкий приоритет)

2. **Обновление DndHelper**: В файле `Helpers/DndHelper.js` добавить логику определения целевой секции долгого ящика при drop и установки соответствующего приоритета элемента, учитывая текущую раскладку (определяемую через `UsersStore.selected`).

3. **Обратная совместимость**: При перетаскивании элемента из одного долгого ящика в другой (в старом режиме) сохранять текущий приоритет элемента или устанавливать `1` (средний) по умолчанию.

### Изменения в сторах

Для поддержки новой функциональности потребуются следующие изменения в сторах:

1. **Хранилища элементов (ItemsStore, ItemsMultiStore)**:
   - Добавить computed-свойства для фильтрации элементов по приоритету bucket (`lowPriorityItems`, `mediumPriorityItems`, `highPriorityItems`)
   - Обновить методы загрузки и обновления элементов для корректной работы с геттером/сеттером `priority`

2. **PeriodsStore и IntervalItem**:
   - Изменить логику распределения элементов в bucket-интервалы для учета приоритета `priority`
   - В методе `reinitPeriods()` для bucket-интервалов создать три периода вместо одного (для отображения трех секций)

### Механизм переключения между старым и новым режимами

Для обеспечения плавного перехода и обратной совместимости реализовать следующий механизм переключения:

1. **Добавление глобального переключателя**: В `LayoutStore` добавить observable-свойство `useSplitBucket` типа boolean со значением по умолчанию `false` (старый режим).

2. **Логика выбора режима отображения**:
   - Когда `useSplitBucket = false`: использовать текущую реализацию с одним долгим ящиком для всех элементов без даты
   - Когда `useSplitBucket = true`: использовать новую реализацию с тремя секциями долгого ящика, разделенными по приоритету

3. **Элемент управления в интерфейсе**: Добавить переключатель в панель настроек или хедер приложения, позволяющий пользователю переключаться между старым и новым режимами отображения долгого ящика.

4. **Сохранение предпочтений**: Значение `useSplitBucket` сохранять в локальное храниличе через существующий механизм `saveOption`/`loadOption` в `LayoutStore`.

5. **Обратная совместимость**: При переходе со старого режима на новый:
   - Все существующие элементы без даты получают приоритет `priority = 1` (средний) по умолчанию (через сеттер, который выполнит инициализацию)
   - Пользователь может вручную изменить приоритет элементов через интерфейс (drag-and-drop между секциями или контекстное меню)

6. **Переход с нового режима на старый**:
    - Все элементы отображаются в едином долгом ящике независимо от их приоритета
    - Приоритеты элементов сохраняются в данных для возможного будущего возврата к новому режиму

Эти изменения позволяют реализовать требуемую функциональность разделения долгого ящика на три приоритета с возможностью выбора варианта отображения (внизу или справа, определяемого текущим выбором пользователя) и плавным переключением между старым и новым режимами для обеспечения обратной совместимости.
